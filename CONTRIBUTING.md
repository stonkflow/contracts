# CONTRIBUTING

Документ для разработчиков, которые вносят изменения в protobuf-контракты stonkflow.

Если вы не работали с Kafka, начните с раздела "Минимальная база".

## Минимальная база: Kafka и protobuf-контракты

### Что такое Kafka topic

Topic — канал сообщений:
- продюсер пишет сообщения в topic
- консюмер читает сообщения из topic

Нас интересует формат `value`, он описан в `.proto`.

### Что такое контракт в этом репозитории

Контракт — это `.proto` файл в `proto/`, который фиксирует структуру сообщений.
Изменение контракта влияет на продюсеров и консюмеров этого события.

### Что такое совместимость (breaking check)

`buf breaking` проверяет, не ломает ли новая версия `.proto` старых потребителей.
В этом проекте breaking может быть допустим, если это осознанный массовый rollout.

## Идеология изменений в stonkflow

В проекте принята стратегия **big-bang rollout**:
- контракт может меняться несовместимо
- продюсеры и консюмеры обновляются одновременно в инфраструктуре
- если часть сервисов не обновилась и падает на новом контракте, это ожидаемое и явное поведение

Мы не поддерживаем параллельные ветки контрактов `v1/v2` для одного события в рамках этой стратегии.

## Что меняется в этом репозитории

Вы меняете:
- protobuf-контракты в `proto/`
- сгенерированные DTO в `gen/`
- документацию

Вы не меняете здесь:
- бизнес-логику сервисов
- настройки Kafka-кластера

## Пошаговый процесс изменения контракта

1. Создайте ветку от `main`.
2. Измените нужный `.proto` в `proto/`.
3. Запустите проверки:

```bash
make lint
make breaking
```

4. Сгенерируйте DTO:

```bash
make generate
```

Важно:
- генерация DTO выполняется только вручную разработчиком
- CI/CD не выполняет `buf generate` и не обновляет `gen/` автоматически

5. Убедитесь, что изменения в `gen/` закоммичены.
6. Если изменение несовместимо, явно пометьте это в PR:
- добавьте в заголовок или описание маркер `BREAKING CHANGE`
- кратко опишите, какие сервисы должны быть обновлены одновременно
- укажите окно выката
7. Обновите таблицу контрактов в `README.md` при добавлении нового контракта.
8. Откройте PR с описанием изменений.

## Правила изменения protobuf

Рекомендуется:
- добавлять новые поля с новыми номерами
- не менять смысл существующих полей без явного описания в PR
- не переиспользовать номера удаленных полей
- обновлять документацию при изменении семантики

Допускается:
- удаление полей
- изменение типов полей
- другие несовместимые изменения

Условие для допуска breaking-изменения:
- оно заранее помечено как `BREAKING CHANGE`
- есть подтвержденный план синхронного обновления продюсеров/консюмеров

## Релиз и распространение контрактов

Контракты распространяются через GitHub Release:
- создается тег `vX.Y.Z`
- в релиз автоматически прикладывается `proto-vX.Y.Z.tar.gz`
- внешние команды используют этот архив для генерации DTO в своих языках

## CI ожидания для PR

Обычно PR должен проходить:
- `buf lint`
- `buf breaking`

Если `buf breaking` падает и это intentional breaking:
- это фиксируется в PR как `BREAKING CHANGE`
- решение о merge принимается maintainers в рамках согласованного окна big-bang rollout

## Чеклист перед PR

- [ ] Изменения в `proto/` минимальны и обоснованы
- [ ] `make lint` проходит
- [ ] `make breaking` проходит (или PR явно помечен как `BREAKING CHANGE`)
- [ ] `make generate` выполнен
- [ ] Изменения в `gen/` добавлены в PR
- [ ] README/документация обновлены
